Executive Summary

During the February 2026 reporting cycle, an independent audit of the response time data pipeline identified two significant errors in how response time data was being processed before it reached the monthly report. Both errors have been corrected, all historical data has been recalculated, and the corrected values have been independently verified. This report explains what was wrong, how it was fixed, and what the corrected values now show.

Bottom line: Response time values previously shown in Power BI were inaccurate for the Routine call category - averaging as low as 48 seconds dispatch-to-scene - because internal officer activities (meal breaks, patrol details, training, TAPS assignments, etc.) were included in the calculation. After correction, Routine call response times are consistent with operational expectations. Emergency and Urgent values were not materially affected.

What Was Wrong

Two separate errors were found and corrected in the automated data processing script that produces the response time statistics for Power BI.

Finding 1 Administrative Activities Included as Calls for Service

The processing script was not filtering out officer self-initiated administrative activities before computing averages. Activities such as meal breaks, coffee breaks, patrol checks, TAPS operations, directed traffic details, training assignments, court duties, vehicle maintenance, and task assignments were all treated the same as genuine public calls for service.

These administrative entries show near-zero response times because the officer is already at a fixed location or on an assigned task - there is no actual travel or dispatch processing occurring. Including them in the average pulled the Routine category dramatically downward. Over 57–61% of all records in the dataset were administrative entries of this type.

Finding 2 First-Arriving Officer Not Always Selected

For incidents where multiple officers respond, the script was not consistently selecting the first officer to arrive on scene. Instead, whichever officer's record appeared first in the source data file was used. This affected approximately 28% of all incidents. After correction (sorting by arrival time before deduplication), Emergency and Urgent response times were reduced by 6–20 seconds - a meaningful but not dramatic change.

Finding 3 Routine Category Distribution (Known Characteristic)

After correcting both errors above, an additional data characteristic was identified: the Routine "Dispatch to On Scene" metric still shows a wide distribution because self-initiated enforcement activity - specifically motor vehicle stops and field contacts - remains in the dataset. In these cases, the officer has already stopped a vehicle before calling it in to dispatch, so the recorded travel time is near zero. This is not an error; it reflects real operations. It is disclosed in the methodology note on the report page.

Why the Corrected Values Are Defensible

Independent peer review: The corrected data and methodology were reviewed by two independent AI systems (Sonnet and Opus) on February 26, 2026. Both confirmed that the fixes are correctly implemented and that the corrected values follow expected operational patterns.

The corrected methodology now applies the following rules, consistently, to all data:

RuleWhat It DoesWhy It Matters
Administrative exclusion 92 officer activity types excluded (meal breaks, TAPS, training, directed details, court, etc.) Prevents internal officer assignments from being counted as public calls for service
First-arriving officer For multi-unit responses, only the first officer to arrive on scene is counted Reflects true department response capability; consistent with national standards (NFPA 1710)
Outlier filter Computed times outside 0–10 minutes are excluded Removes data entry errors and atypical delays that do not reflect typical operations
Call type mapping Calls without a priority code are matched against a 664-entry reference table Ensures 2024 data (where CAD priority was unpopulated in <1% of records) is fully classified
Rolling 13-month window Each monthly report shows only the 13 months prior to the report date Trend line reflects recent performance; historical reports remain stable and reproducible

Do the Corrected Values Make Operational Sense?

The corrected values follow a logical pattern consistent with how each call type is handled operationally:

Emergency calls have the longest total response time (approximately 4:41–5:03 from call receipt to on scene) because they require dispatcher coordination - verifying the location, gathering situational information, and notifying the appropriate units - before officers can be released. The dispatch processing phase alone averages approximately 2:48 for Emergency calls.

Urgent calls have shorter total times (approximately 3:05–3:44) because they frequently arrive through structured channels such as alarm companies, where pre-formatted information reduces dispatcher processing time.

Routine calls have the shortest total times (approximately 2:43–3:22) reflecting that dispatchers assign the next available nearby officer without the coordination overhead required for higher-priority calls. The Routine category also includes self-initiated enforcement activity such as motor vehicle stops, where the officer is already present - this is disclosed in the report footnote.

The large difference between raw records and deduplicated records reflects the fact that each incident in CAD generates one row per responding officer. Only the first-arriving officer's row is retained. The further reduction from deduplication to admin exclusion reflects the volume of internal officer activities logged in CAD (57–61% of deduplicated records). This is consistent with officers spending more time on administrative functions, directed patrol, and self-initiated activity than on dispatched public calls.
